1. دورك في المشروع

دورك الأساسي إنك ماسك قلب NOVAIS: Laravel API، تصميم الجداول والعلاقات، توليد المحتوى بالـ AI، صلاحيات المستخدمين والأدمن، الدفع، الإشعارات، والتصدير. يعني أي شاشة في الويب أو الموبايل بتكلم الباك غالبا بتعدي على شغل أحمد.

What to say in defense: "أنا مسؤول عن الباك إند والداتابيز والـ AI integration. ركزت إن نفس الـ API يخدم الويب والموبايل والديسكتوب، وإن المحتوى يتولد بشكل منظم حسب blueprint مش مجرد نص عشوائي."

2. فكرة NOVAIS

NOVAIS منصة تعليمية بتستخدم AI لتوليد منتجات تعليمية مختلفة: كورسات، كتب، بنك أسئلة، امتحانات، مراجعات، خطط دروس، ومشروع تخرج. المستخدم يختار النوع واللغة والمستوى، والباك يطبق قواعد النوع ويولد outline ثم دروس/محتوى عند الطلب.

  • الويب والموبايل والديسكتوب كلهم عملاء لنفس Laravel API.
  • الأدمن يغير إعدادات المنصة والـ blueprints من الداشبورد.
  • الدفع مدعوم عن طريق Paymob وطلبات دفع Offline.

3. المعمارية العامة

Clients

React web، Flutter mobile، و Electron desktop wrapper. كلهم بيبعتوا requests للـ API.

API Layer

Routes في backend/routes/api.php بتقسم public/auth/admin/user endpoints.

Domain Services

CourseService و DeepSeekService و PaymobService شايلين المنطق الثقيل.

Database

Models ومigrations للعلاقات: users, courses, lessons, payments, subscriptions, blueprints, settings.

5. تدفق البيانات من أول الضغط لحد المحتوى

  1. العميل يجيب /platform-settings و /content-blueprints.
  2. المستخدم يختار blueprint ويدخل fields زي topic/language/level.
  3. الويب أو الموبايل يبعت POST /generate-course.
  4. CourseService يتحقق من الاشتراك، اللغة، السلامة، والـ blueprint.
  5. DeepSeekService يبني prompt ويرجع JSON.
  6. CurriculumValidator يطبع ويصلح structure لو ناقص.
  7. المستخدم يراجع outline ثم يحفظ عن طريق POST /courses.
  8. الدروس التفصيلية تتولد lazy عند فتح lesson endpoint.

6. الداتابيز والعلاقات

الداتابيز مش جدول courses وخلاص. في طبقات تمثل المستخدم، الاشتراك، المحتوى، الدفع، الإشعارات، والإعدادات.

Learning

courses, lessons, quizzes, quiz_questions, certificates.

Business

plans, payments, subscriptions, offline_payment_requests.

Configuration

platform_settings, content_blueprints, CMS pages/blogs/social links.

Engagement

app_notifications, notification_devices, notes, chat history, shares.

Doctor may ask: ليه خزنت blueprint metadata جوه course؟ عشان المنتج يفضل عارف نوعه وعناوينه حتى لو إعدادات الأدمن اتغيرت بعدين.

8. الأمان والصلاحيات

  • JWT كل protected endpoint تحت auth:api.
  • Admin Middleware routes الأدمن تحت is_admin.
  • Ownership المستخدم ميقدرش يفتح كورس/إشعار/طلب دفع مش بتاعه.
  • Paymob HMAC webhook بيتحقق بتوقيع قبل تفعيل الاشتراك.
  • Validation كل store/update فيه Laravel validation.
  • Public Sanitization public settings ما بتطلعش secrets أو admin-only keys.

9. الذكاء الاصطناعي والـ Blueprints

DeepSeekService بيبني prompts حسب نوع المنتج. لو blueprint موجود، prompt بياخد schema وتعليمات النوع بدل template واحد للكورسات. الخدمة بتعمل retries، parsing للـ JSON حتى لو AI رجعه جوه code fence، وبتسجل usage/logs.

نقطة قوة: المشروع بقى content-product platform مش course generator بس. ContentBlueprint فيه defaults لأنواع زي normal course, academic course, question bank, exam builder, book, story, graduation project.
حدود مهمة: AI output محتاج مراجعة بشرية. الكود بيحاول يطبع structure ويمنع محتوى خطر، لكنه مش ضمان أكاديمي مطلق.

10. الدفع والاشتراكات

Paymob Online

PaymentController يبدأ checkout، و PaymobService يتعامل مع auth/order/payment key. التفعيل الحقيقي يتم من webhook بعد HMAC ومطابقة amount/order.

Offline Payments

Vodafone Cash/InstaPay request مع رقم العملية أو صورة proof. الأدمن يوافق أو يرفض، والموافقة تعمل subscription وتحدث user role.

Doctor may ask: ليه callback مش كفاية؟ لأن callback ممكن يتزور من المتصفح. القرار النهائي لازم webhook موقع من Paymob.

11. لوحة الأدمن

الأدمن يقدر يدير users, courses, paid users, plans, offline payments, CMS pages/blogs, platform settings, content blueprints, notifications, media. الباك عامل routes منفصلة للأدمن ومحمية بصلاحيات.

12. التصدير والفواتير

المشروع يستخدم DomPDF لتصدير PDF وفواتير الدفع. يوجد endpoints للتصدير للـ PDF/PPT في الويب، والموبايل يعرف endpoints التصدير في الكود. في worktree الحالي فيه شغل غير مكمت على document export view، لذلك في الدفاع نقول: "التصدير موجود، وفي تحسينات جارية لتفريق شكل المستندات عن الكورسات".

13. الاختبارات والتحقق

  • Backend tests مذكور في docs إنها مرت: 93 tests / 300 assertions.
  • اختبارات مهمة: auth, authorization, payments security, offline payments, platform config, blueprints, notifications.
  • React build و Flutter analyze/test/build و Electron build مذكورين كناجحين في readiness docs.
  • Composer audit بدون advisories حسب docs/الفحص السابق، بينما npm عنده moderate advisories لا توقف build.

15. What To Say In Defense

"NOVAIS backend معمول كـ API-first architecture. نفس Laravel API بيخدم React web وFlutter mobile وElectron desktop. ركزت على إن الـ AI يرجع structured data مش كلام حر، وإن كل business flow له validation وصلاحيات واختبارات."

Doctor May Ask

1. المشروع ده course generator بس؟
لا. دلوقتي فيه content blueprints لأنواع تعليمية متعددة، والكورس مجرد نوع منهم.
2. فين نقطة دخول الـ API؟
backend/routes/api.php، ومنه routes بتروح للcontrollers والخدمات.
3. ليه استخدمت Services؟
عشان controller يفضل خفيف، والمنطق زي AI/payments/course generation يبقى قابل للاختبار والصيانة.
4. إزاي بتحمي admin routes؟
بتحتاج JWT ثم middleware is_admin يتحقق من دور المستخدم.
5. إزاي تمنع user يشوف كورس غيره؟
الcontrollers بتفلتر حسب user_id أو تتحقق من ownership قبل إرجاع البيانات.
6. ليه JWT؟
مناسب لـ API يخدم web/mobile/desktop بدون session server-side تقليدية.
7. إزاي Google login داخل في السيستم؟
باستخدام Socialite وroutes خاصة بـ Google OAuth، وبعد النجاح بنرجع token.
8. إيه وظيفة CheckDevice؟
يبعت ويتابع device id لتقليل إساءة الاستخدام وربط بعض flows بالجهاز.
9. إزاي الـ AI بيرجع JSON؟
prompt صارم + parser ينضف code fences/trailing commas + validator يطبع structure.
10. لو AI رجع ناقص؟
CurriculumValidator يحاول normalize ويكمل fallback lessons/sections عند الحاجة.
11. إيه content blueprint؟
تعريف لنوع محتوى: اسمه، schema الحقول، prompt instructions، display terms، والـ output sections.
12. ليه blueprints في الداتابيز؟
عشان الأدمن يغير الأنواع والتعليمات بدون تعديل كود ونشر جديد.
13. إزاي بتحمي prompt instructions؟
public endpoint بيرجع نسخة sanitized؛ الأدمن فقط يرى full config/instructions.
14. إيه safety validation؟
فلترة محلية لكلمات خطرة + AI moderation قبل generation حسب الإعدادات.
15. هل safety perfect؟
لا، هي طبقة دفاع. لازم نقول إنها تقلل المخاطر وليست ضمان مطلق.
16. إيه Paymob flow؟
create pending payment، طلب checkout، ثم webhook موقّع يفعّل الاشتراك.
17. ليه تتحقق من amount في webhook؟
عشان مايتفعلش اشتراك بسعر غلط أو order مختلف.
18. إيه HMAC؟
توقيع من Paymob بنحسبه بنفس السر ونقارنه بـ hash_equals.
19. إيه offline payment؟
طلب دفع محلي برقم عملية أو صورة إثبات، والأدمن يراجعه ويوافق/يرفض.
20. إزاي تمنع duplicate offline requests؟
الكود يمنع pending duplicate لنفس user/plan/cycle/method.
21. إيه platform settings؟
إعدادات عامة للمنصة: branding، features، payments visibility، export flags، وغيرها.
22. ليه public settings sanitized؟
عشان frontend يحتاج إعدادات، لكن مش لازم ياخد secrets/admin keys.
23. إيه notification devices؟
تسجيل أجهزة المستخدمين مع optional push token. حاليا in-app جاهز، FCM native محتاج إعداد Firebase.
24. هل desktop له backend مختلف؟
لا، Electron يشغل نفس React bundle ويكلم نفس API.
25. إيه أهم اختبار payment؟
اختبارات security/webhook/offline approval لأنها تحمي money flow.
26. إزاي بتتعامل مع أخطاء AI؟
timeouts/retries/logging ورسائل خطأ مناسبة، ولو JSON بايظ بنحاول parse/repair.
27. ليه lazy lesson generation؟
عشان نقلل وقت وتكلفة إنشاء outline كامل، ونولد التفاصيل لما المستخدم يفتح الدرس.
28. إيه role بتاع subscription؟
بيحدد plan limits وpremium role والمدة والسعر، ويمنع إنشاء محتوى فوق الحد.
29. إزاي الفواتير بتتعمل؟
بعد payment success بيتولد invoice PDF ويتبعت email، مع logging لو الإيميل فشل.
30. إيه أهم limitation؟
الـ AI محتاج مراجعة، وPR الحالي draft، وnative push/live Paymob production مش مثبتين بالاختبارات.
31. ليه Laravel validation مهم؟
بيمنع inputs ناقصة أو مضرة قبل الوصول للbusiness logic.
32. إزاي course مربوط بالlessons؟
Course model له lessons، والـ save flow ينشئ placeholders ثم المحتوى يتولد عند الطلب.
33. إيه chat endpoint؟
مساعد داخل الكورس يستخدم history والرسالة ويرجع رد من AI.
34. إزاي share links؟
backend يعمل token public share، وendpoint public يعرض المحتوى المسموح.
35. إيه الفرق بين platform-config وplatform-settings؟
الاتنين يخدموا configuration للعميل، مع تركيز على public sanitized config للfrontends.
36. إزاي تعرف إن الباك شغال؟
من tests، migration readiness، API inventory، وتجارب clients اللي بتبني على نفس endpoints.
37. إيه file لازم تفتحه لو route مش شغال؟
ابدأ بـ routes/api.php ثم controller ثم service ثم test.
38. هل في caching؟
نعم settings/config public cached، والموبايل عنده GET cache client-side.
39. إيه سبب وجود AIProviderInterface؟
عشان نفصل التطبيق عن مزود AI ونقدر نبدل DeepSeek أو نضيف provider لاحقا.
40. لو الدكتور قال المحتوى غلط أكاديميا؟
نوضح إن النظام بيولد draft structured، والـ human review جزء من الاستخدام المسؤول.
41. إزاي تتكلم عن PR #6؟
بصراحة: PR draft فيه تحسينات blueprints/product terminology، ولسه محتاج مراجعة نهائية قبل merge.
42. إيه أكبر contribution ليك؟
تحويل الفكرة من صفحات بتطلب AI لنظام backend منظم فيه صلاحيات، بيانات، دفع، AI، وadmin control.

Common Mistakes

  • ما تقولش "AI مضمون 100%"؛ قول structured + validated + needs review.
  • ما تقولش "Paymob production tested" لو مفيش دليل live production.
  • ما تخلطش بين callback وwebhook.
  • ما تنساش تقول إن public blueprints لا تكشف prompt instructions الكاملة.
